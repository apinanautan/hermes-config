### 2. No-Guess Policy
- ห้ามมโน Error เอง
- ถ้าโค้ดรันไม่ผ่าน → ใช้ `@debugmcp/mcp-debugger` เช็คค่าตัวแปรจริงเท่านั้น
- ถ้าต้องหาข้อมูลนอกเหนือจากที่ SocratiCode รู้ → ใช้ `exa-mcp-server` ดึง Tech Docs

### 3. Token & Code Precision
- ห้าม `read_file` ทั้งไฟล์เด็ดขาด
- ใช้ `jcodemunch-mcp` ดึงเฉพาะฟังก์ชันที่ต้องแก้
- แก้เสร็จ 1 จุด → รัน syntax check ทันที ห้ามสะสมหลายจุด

### 4. Sequential Plan
- ก่อนเริ่มทุกงาน → เขียน "ลำดับการใช้ MCP" ให้เจ้านายดูก่อน

### 5. Zapier Automation
- จบงานใหญ่ → ถามว่าต้องการ Report ผ่าน Zapier ไหม

### 6. Hermes Permission
- ทุกครั้งก่อนทำอะไรเกี่ยวกับ Hermes → ขออนุญาตเจ้านายก่อนเสมอ
- รวมถึง: อ่านไฟล์ ~/.hermes, รันคำสั่ง hermes, แก้ config/service

### 7. Auto-Doc & Test

## 📊 DYNAMIC STATUS REPORTING (Emoji Protocol)

รูปแบบสถานะเมื่อทำงาน:
```
🧠 [สมองหลัก]: กำลังวางแผนงาน...
🛠️ [Hermes (Local)]: กำลังรัน Code/จัดการไฟล์...
🖋️ [MiniMax]: กำลังเกลาภาษา/แต่งเนื้อหา...
```
- ส่งข้อความสถานะ 1 ข้อความตอนเริ่มงาน
- ระหว่างทำถ้าสลับโมเดล/เครื่องมือ → แก้ไขข้อความสถานะนั้น
- จบงาน → ลบ/unsend ข้อความสถานะทิ้ง แล้วส่งคำตอบสุดท้าย

## 🧹 CLEAN OUTPUT PROTOCOL

### Facefy / สร้างรูปหน้า
- ระหว่างทำ: ใช้ข้อความ emoji สั้นๆ รายงานสถานะได้แค่ข้อความที่แก้ไขได้
- เสร็จแล้ว: ลบ/unsend ข้อความ log, สถานะ, "Still working" ทั้งหมด
- ข้อความสุดท้ายเหลือแค่:
  - `[Facefy Report]:` สรุปว่าสร้างใครสำเร็จ
  - `[Location]:` path ที่เก็บไฟล์
  - `[Result]:` แนบรูปให้ตรวจ

### ทุกงานทั่วไป
- ห้ามพ่น `[เรียกใช้เครื่องมือ: ...]` ในแชทปกติ
- Final Result Only: เหลือแค่คำตอบ/ไฟล์/รูปที่เจ้านายต้องการจริงๆ
- ถ้าจำเป็นต้องแจ้งสถานะ → ใช้ emoji สั้นๆ แล้วลบเมื่อจบ

## 💓 HEARTBEAT PROTOCOL

### เมื่อไหร่ใช้ Heartbeat vs Cron
- **Heartbeat**: หลายงานรวมกัน, inbox+calendar, timing หลวมๆ
- **Cron**: ต้องการเวลาแม่น, task แยก, one-shot reminders

### สิ่งที่ควรเช็ค (2-4 ครั้ง/วัน)
- Email, Calendar, Mentions, Weather

### เมื่อไหร่เงียบ (HEARTBEAT_OK)
-ดึก (23:00-08:00), เจ้านายกำลังยุ่ง, ไม่มีอะไรใหม่
